# 비동기의 본질

## 1. 동기(Sync) 방식의 흐름
- 작동: 메인 스레드가 네트워크 함수를 호출하면, 데이터가 올 때까지 그 함수 안에서 멈춰 기다린다.

- 문제: 호출한 함수가 끝나기 전까지는 메인 스레드는 다음 행동(코드)를 실행하지 못한다. [Thread.md](./Thread.md)에서 배웠던 화면 멈춤의 원인이다.

## 2. 비동기(Async) 방식의 흐름
- 작동: 메인 스레드가 작업을 지시하고, **즉시 제어권을 반환받아** 다음 행동(코드)로 넘어간다

- 핵심: 작업은 OS나 다른 스레드가 뒤에서 따로 처리하고, 메인 스레드는 계속해서 사용자의 터치 피드백을 받거나 애니메이션을 그린다.

## 비동기 통신의 남은 일
- 상황: 메인 스레드가 백그라운드 스레드에게 "네트워크 통신해" 라고 일을 던진 뒤, 제어권을 즉시 반환받아 화면을 그리고 있는다.

- 문제: 백그라운드 스레드의 일이 다 끝났을 때, **그 결과물을 어떻게 메인 스레드에게 안전하게 전달할 것인가?**

## 방식 A: 콜백(Callback)
가장 고전적인 방식. 메인 스레드가 일을 시킬 때, 데이터만 요구하는 것이 아니라 행동 요령을 같이 넘겨준다.
- 작동: 백그라운드 스레드는 통신이 끝나면, 메인 스레드가 쥐여준 행동 요령(콜백 함수)를 보고 그대로 실행하여 화면을 바꾼다.
- 단점: A통신 끝나고 B통신, B 끝나고 C통신을 연달아서 해야할 때 발생한다. 이와 같이 코드가 우측으로 끝없이 파고 들어가는 현상(Callback Hell)이 발생하여 가독성이 최악이 된다.

## 방식 B: 퓨처(Future/ Promise)
현대 프로그래밍(dart, kotlin)에서 콜백 지옥을 해결하기 위해 사용하는 핵심 규약이다.
- 작동: 메인 스레드가 일을 지시하면, 백그라운드 스레드는 즉시 Future를 메인 스레드에게 던져준다. 위 Future는 현재 비어있지만, 나중에 결과가 나온다면 위 Future에 담길 것을 약속한다는 뜻이다.
- 상태 변화
    - **`대기(Pending)`**: 통신 중이라 Future가 비어있는 상태
    - **`완료(Fulfilled)`**: 통신이 성공하여 Future에 결과값이 담긴 상태
    - **`거부(Rejected)`**: 통신에 실패하여 Future 안에 에러 메시지가 담긴 상태

## 3. Aysnc/Await
Future를 받았을 때, 콜백 함수를 쓰지 않고 **마치 동기식 코드처럼** 깔끔하게 읽히도록 만드는 현대적 문법이다.

- **`await`**: 네트워크 함수 앞에 이 키워드를 붙이면, Future에 데이터가 완전히 채워질 때까지 해당 줄에서 코드 실행을 일시 정지한다.

- **`async`**: 함수 내부에 `await`을 사용하려면, 그 함수가 비동기로 동작한다는 것을 OS에 알려주기 위해 함수 이름 옆에 `async`를 붙여주어야 한다.

- **`논블로킹`**: `await`이 코드를 멈추게 하더라도, **메인 스레드를 멈추게** 하는 것은 아니다. 해당 함수의 실행만 잠깐 멈춰두고, 메인 스레드는 즉시 제어권을 넘겨받아 화면을 그리거나 터치를 받는 본연의 일을 계속한다.

## 확인 퀴즈
- 상황: 기획 중인 '습관 교정 앱'에서 다음 3단계 네트워크 통신을 진행한다.

1. 서버에서 계정 정보 가져오기
2. 계정 정보를 바탕으로 오늘의 습관 리스트 가져오기
3. 습관 리스트를 바탕으로 달성률 데이터 가져오기

- 질문: 1번 통신은 성공했지만, 2번(습관 리스트)에서 서버 오류로 통신에 실패했다. 빈 상자(Future) 개념을 적용했을 때, 3번 통신은 어떻게 처리되어야 하며, 메인 스레드는 상자의 상태 변화를 이용해 사용자에게 이 상황을 어떻게 보여주는 것이 가장 논리적인가?

내 답: 2번에서 거부처리가 된 상태를 받아와서 3번도 마찬가지로 거부 상태가 되어야한다. 메인 스레드에서는 앞선 2번에서 계정정보에서 오늘의 습관을 못받아와서 이후 작업이 진행되지 않았다는 에러 메시지와 함께 재시도를 하는 UI등으로 사용자에게 보여주는 게 좋을듯하다.
