# CPU 레지스터와 명령어 실행 원리

## 1. CPU의 단기 기억 상실증과 레지스터
CPU는 연산속도는 엄청나게 빠르지만, 메모리에 갔다 오는 시간을 기다려주지 않는다. 그래서 자기 바로 옆에 아주 작은 용량의 초고속 메모리를 두는데, 이것이 **`레지스터`** 다.

- **`PC(Program Counter)`** : 다음에 실행할 명령어의 주소를 가르키는 `이정표`이다.
- **`IR(Instruction Register)`** : 메모리에서 방금 가져온 `명령어`를 담는 곳이다.
- **`ALU(산술논리연산장치)`** : 실제 덧셈, 뺄셈, 논리 연산을 수행하는 `계산기`이다.

## 2. 명령어 실행 주기
프로그래머가 작성한 코드가 CPU에서 처리되는 4단계 과정
1. **`Fetch(인출)`** : 메모리에서 명령어를 가져와 IR에 넣는다.
2. **`Decode (해독)`** : 제어장치가 받아온 명령어를 해석한다.
3. **`Execute (실행)`** : ALU가 계산기를 두드려 연산한다.
4. **`Write-back (쓰기)`** : 결과를 레지스터나 메모리에 저장한다.

# 캐시 메모리와 참조 지역성 (성능 최적화 핵심)

## 1. 캐시 메모리 (Cache Memory)
CPU와 Ram 사이의 속도 차이를 극복하기 위한 중간 도구이다.
- L1, L3, L3캐시: CPU와 가까울수록 빠르고 비싸고, 용량이 적다.
- **`캐시 히트(Cache Hit)`** : 필요한 데이터가 캐시에 있어서 바로 가져오는 경우 (초고속)
- **`캐시 미스(Cache Miss)`** : 캐시에 없어서 느린 Ram까지 갔다 와야 하는 경우 (성능 저하)

## 2. 참조 지역성
캐시가 효율적으로 작동하는 이유는 프로그램이 가진 두 가지 성질 때문이다.
- **`시간 지역성`** : 한 번 쓴 데이터는 곧 또 쓸 가능성이 높다. (ex. 반복문의 변수)
- **`공간 지역성`** : 지금 쓴 데이터 주변의 데이터도 같이 쓸 가능성이 높다. (ex. 배열, 리스트)

# 가상 메모리와 페이징 (OS의 메모리 관리)

## 1. 가상 메모리
앱이 실제 Ram 주소를 건드리지 못하게 OS가 가짜 주소 공간을 주는 기술이다.
- 단점: SSD속도는 Ram 입장에서 매우 느리기 때문에 속도차이가 있다.
- 장점: 실제 Ram보다 더 큰 프로그램을 돌릴 수 있고, 앱끼리 메모리 침범을 막는다.
- 어떻게?: OS단에서 앱에게 가짜 주소(가상 메모리)를 부여한다. 두 앱은 진짜 Ram 주소를 모른다. 그리고 두 앱이 부여 받은 주소에 데이터를 쓰겠다고 CPU에 요청을 하면, 이때 `MMU(메모리 관리 장치)`라는 하드웨어가 개입하여 OS가 가지고 있는 `페이지 테이블`이라는 번역기를 사용한다.
- 만약 실제 주소에 접근하려고 악의적인 코드를 짠다하더라도, MMU를 거쳐야하는데 애초에 길 자체가 막혀있어 접근을 못한다. 그리고 OS단에서  `세그멘테이션 폴트(Segmentation Fault)`라는 에러를 띄우고 앱을 즉시 강제 종료 한다.

## 2. 페이징
메모리를 일정한 크기로 잘라서 관리하는 기법이다.
- `Page Fault`: 실행하려는 코드가 Ram에 없어서 저장자이에서 가져와야 할 때 발생한다. 이때 앱이 일시적으로 느려진다.
