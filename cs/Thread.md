# Thread와 UI 병목 현상

## 1. 프로세스와 스레드의 개념
- **`프로세스`**: 실행 중인 앱 그 자체. 스마트폰에서 카카오톡 아이콘을 눌러 실행하는 순간, OS는 카카오톡이라는 '프로세스'를 하나 생성하고 작업 공간(메모리)을 할당해준다.
- **`스레드`**: 프로세스 내부에서 실제로 일을 하는 '일꾼'이다. 하나의 앱(프로세스) 안에는 여러 명의 일꾼(스레드)가 동시에 존재할 수 있다. 이를 **멀티 스레딩**이라고 한다.

## 2. 모바일 앱의 황금률: 메인 스레드의 독재
안드로이드와 플러터 모두 앱이 켜지면 기본적으로 단 1명의 스레드만 생성된다. 이를 **메인 스레드** 혹은 **UI 스레드**라고 부른다.
- 역할: 위 스레드의 유일하고 가장 중요한 임무는 **"화면에 UI를 그리고, 사용자의 터치 입력을 받는 것"** 이다. 1초에 60번씩 화면을 새로고침(60hz)하여 부드러운 애니메이션을 만들어낸다.

## 3. 왜 앱 화면이 멈추는가? (ANR: Application Not Resopnding)
주니어 개발자들은 이 메인 스레드에게 화면 그리는 일 외에 다른 무거운 일을 시키는 실수를 한다.
- 잘못된 예: 사용자가 '다운로드' 버튼을 누른다. ->  개발자가 1GB짜리 파일을 다운로드 하는 작업을 메인 스레드에 시킨다.
- 결과: 메인 스레드는 파일 다운로드가 끝날 때까지 다른 일을 못한다. 즉, 화면 새로고침하지도 못하고, 사용자 터치 감지도 못한다.
- 화면이 굳어버리면 안드로이드 OS는 해당 앱이 고장났다고 판단하고, **"앱이 응답하지 않습니다(ANR)"** 라는 경고창을 띄우고 앱을 강제로 죽여버린다.

## 4. 해결책: 백그라운드 스레드(Background Thread)와 비동기(Asynchronous)
무거운 작업(네트워크 통신, 대용량 DB 저장, 파일 다운로드)은 메인 스레드가 직접 하게 두면 안된다.
- 백그라운드 스레드를 하나 더 고용해서 "백그라운드에서 파일 다운로드만 해!" 하고 시킨다.
- 메인 스레드는 작업을 던져놓고 곧바로 다시 화면을 그리는 터치를 받는 일을 계속 한다.
- 백그라운드 스레드가 다운로드를 끝내면 메인 스레드에게 다 됐다고 알려주고, 메인 스레드에선 받은 결과를 바탕으로 UI로 보여준다.
- 이렇게 메인 스레드를 멈추지 않고 뒤에서 일을 처리하는 방식을 **비동기(Asynchronous)처리**라고 부른다. 

## 확인 퀴즈
- 상황: a가 만든 앱에는 서버에서 10,000개의 게시글 데이터를 가져와서 화면에 목록으로 띄우는 기능이 있다. 서버에서 데이터를 가져오는 데는 약 3초가 걸린다.
- 질문: 만약 이 3초가 걸리는 네트워크 통신 작업을 **메인 스레드**에서 실행하도록 코드를 짰다면, 사용자가 앱에서 게시글 목록 버튼을 눌렀을 때 그 3초 동안 스마트폰 화면에서는 물리적으로 어떤 현상이 발생하며, 그 이유는 무엇인지 **스레드의 역할**을 근거로 서술하시오.

내 답: 메인 스레드에서 해당 작업을 진행하게 되면 다른 작업들을 못하기 때문에 화면이 멈추고, 해당 작업을 진행하는 동안 사용자의 터치 피드백도 반응 못하고, 사용자 네트워크가 좋지 않다면 더 오래 걸릴거고, 오래 걸리면 OS에서 고장났다 판단해서 앱을 강제종료 때린다.
